## DEC-01 · OCR 方案选型

### 1.1 背景

绘本朗读助手的核心功能是将绘本页面图片转换为文字，OCR（光学字符识别）方案的选择直接决定了用户体验质量与研发复杂度。在 MVP 阶段，我们需要评估如何让用户以最低操作成本获得可靠的识别结果。

绘本文字的特点：
- 字体多样（艺术字、手写风格字体常见）；
- 字号偏大（专为幼儿设计，字体清晰）；
- 文字与插图混排（文字区域不固定）；
- 单页文字量较少（通常 10–100 字）。

### 1.2 备选方案

#### 方案 A：手动框选识别
用户在拍照后，用手指框选需要识别的文字区域，系统只识别框选范围内的内容。

**优点：**
- 避免将插图中的装饰性文字误识别；
- 用户可精确控制识别范围；

**缺点：**
- 操作步骤增加（拍照 → 框选 → 识别），门槛高；
- 目标用户（家长带孩子使用）时间紧迫，多步操作体验差；
- 幼儿无法自主完成框选操作；
- 部分绘本文字散落在插图各处，框选难以覆盖。

#### 方案 B：自动全图识别（**最终选择**）
拍照后，系统自动对全图进行 OCR 识别，提取图片中的所有文字。

**优点：**
- 操作极简：拍照后一键触发，无需额外交互；
- 符合「零门槛」核心价值主张；
- 支持幼儿自主操作（点击拍照按钮即可）；
- 讯飞/百度等主流 OCR 引擎对印刷体识别准确率高（≥95%）；

**缺点：**
- 可能识别到页码、版权信息等非故事内容；
- 装饰性图案中的字母可能被误识别；
- 需要后处理逻辑过滤噪声文字（如页码、单个字符）。

#### 方案 C：AI 语义区域检测 + 识别
先用视觉模型检测「故事文字」区域，再对该区域做精细 OCR。

**优点：**
- 精度理论最高，几乎不识别到噪声；

**缺点：**
- 需要训练专属绘本文字检测模型，开发成本极高；
- MVP 阶段无法实现，时间不允许；
- 绘本样式多样，模型泛化能力难以保证。

### 1.3 决定

> ✅ **采用方案 B：自动全图识别**

### 1.4 决策理由

1. **用户体验优先**：目标用户（2–6 岁幼儿家长）对操作便捷性要求极高，任何多余的交互步骤都会显著降低使用意愿。「拍照即朗读」的极简体验是核心竞争力。

2. **幼儿自主使用**：部分使用场景下，孩子需要自己操作。框选操作对幼儿而言完全不可行，自动识别是唯一选项。

3. **误识别可接受**：绘本页面以故事文字为主，页码等噪声文字量极少（通常 1–2 处）。后处理过滤规则（如过滤 <2 字符的孤立识别结果）可将噪声影响降至可接受水平。

4. **研发成本可控**：方案 B 基于成熟的 OCR API（微信云 OCR / 腾讯云 OCR），开发工作量小，符合 MVP 快速验证的节奏。

5. **方案 C 超出 MVP 范围**：定制模型方案需 2–3 个月额外开发，不在 Phase 1 时间范围内，待 Phase 2/3 数据积累后可考虑引入。

### 1.5 影响

| 维度 | 影响说明 |
|---|---|
| **用户体验** | ✅ 操作步骤减少，首次使用无学习成本 |
| **研发工作量** | ✅ 直接调用 OCR API，前端开发工作量约 3 人天 |
| **识别准确率** | ⚠️ 可能存在页码等噪声，需后处理过滤（额外 1 人天） |
| **后续迭代** | 📋 Phase 2 评估是否引入语义区域过滤以提升精度 |
| **成本** | ⚠️ OCR API 按调用次数计费，需监控日均调用量，设置预算告警 |

## DEC-02 · TTS 方案选型

### 2.1 背景

TTS（文字转语音）是绘本朗读助手的另一核心能力，语音质量直接影响儿童的听读体验。幼儿对声音极为敏感，机械感强的 TTS 语音会降低吸引力；同时，儿童友好的音色能增强故事沉浸感，提升产品粘性。

核心评估维度：
- **音色自然度**：儿童能否接受，是否有机械感；
- **儿童专属音色**：是否有针对儿童设计的声线；
- **中文质量**：能否正确处理儿化音、语气词等口语化表达；
- **延迟**：首包返回时间是否满足 P90 ≤ 1 秒；
- **成本**：按字符计费，需评估规模化后的成本可控性；
- **接入复杂度**：SDK 成熟度、微信小程序适配情况。

### 2.2 备选方案

#### 方案 A：微信同声传译（小程序内置）

**优点：**
- 无需第三方 SDK，小程序原生支持，包体积零增加；
- 延迟极低（本地合成）；
- 免费；

**缺点：**
- 音色选择极少（仅通用成人音色）；
- 完全无儿童专属音色；
- 语音质量一般，机械感明显；
- 不支持自定义音色，无差异化竞争力。

#### 方案 B：讯飞开放平台 · 儿童音色（**最终选择**）

**优点：**
- 拥有专属儿童音色（小燕、小宇等）；
- 中文质量业界领先，支持儿化音、停顿自然；
- 提供专为幼儿设计的语调和节奏；
- 微信小程序 SDK 成熟，接入文档完善；
- 支持 SSML 标记语言，可精细控制语速、停顿；

**缺点：**
- 需要集成第三方 SDK，增加主包体积（约 200–300 KB，需分包处理）；
- 按字符计费，大规模使用后成本需关注；
- 网络依赖（在线合成模式），离线需缓存。

#### 方案 C：腾讯云 TTS

**优点：**
- 与微信生态同源，潜在技术协同优势；
- 音色多样；

**缺点：**
- 儿童专属音色数量少于讯飞，且质量评测不及讯飞；
- 儿化音处理不够自然；
- 对于幼儿绘本场景，讯飞产品成熟度更高。

#### 方案 D：字节火山引擎 TTS

**优点：**
- 超写实音色质量优秀；

**缺点：**
- 专为幼儿设计的儿童音色产品线较弱；
- 小程序 SDK 成熟度不如讯飞；
- 定价偏高。

### 2.3 决定

> ✅ **采用方案 B：讯飞开放平台 · 儿童音色**

音色规划：

| 音色编号 | 讯飞音色 ID | 产品名称 | 特点 |
|---|---|---|---|
| V01 | `xiaoyan_child`（男童版） | 🧒 小明 | 活泼明亮，适合冒险故事 |
| V02 | `aisxping`（儿童女声） | 👧 小美 | 甜美温柔，适合童话故事 |
| V03 | `xiaoyan`（标准女声） | 👩 暖妈 | 亲切稳重，适合科普绘本 |

### 2.4 决策理由

1. **儿童音色是核心差异点**：目标用户是 2–6 岁幼儿，儿童友好的声音是产品体验的关键。讯飞是市场上儿童音色最丰富、质量最高的 TTS 服务商，没有更好的替代。

2. **中文质量标杆**：讯飞在中文 TTS 领域拥有 20+ 年积累，对儿化音（"一点儿"）、多音字（"长"读音区分）、语气词的处理在业界处于最高水平，这对绘本朗读场景至关重要。

3. **SDK 接入成熟**：讯飞小程序 SDK 有完整的 WeChat 适配，社区案例丰富，能快速集成，降低研发风险。

4. **成本在可接受范围**：绘本单页文字量通常 10–100 字，MVP 阶段预估月均字符消耗在免费额度（500 万字符/月）内；规模化后可通过本地缓存大幅降低 API 调用次数。

5. **竞品差异化**：选用讯飞儿童音色，能在同类产品中形成明显的听觉差异，提升用户口碑传播意愿（"这个声音好好听"是目标用户的高频评论预期）。

### 2.5 影响

| 维度 | 影响说明 |
|---|---|
| **用户体验** | ✅ 儿童专属音色显著提升沉浸感和用户满意度 |
| **主包体积** | ⚠️ 讯飞 SDK 约 250 KB，需放入分包，不计入主包 2MB 限制 |
| **研发工作量** | SDK 接入约 3 人天，音色试听 UI 约 2 人天 |
| **API 成本** | ⚠️ 规模化后按字符计费，需建立用量监控和预算告警机制 |
| **离线支持** | 📋 TTS 音频需缓存到本地（与 DEC-03 联动），避免重复 API 调用 |
| **合规** | 📋 需与讯飞签订数据处理协议，确保用户文本数据处理合规 |

## DEC-03 · 离线缓存方案

### 3.1 背景

绘本阅读场景常常发生在网络不稳定的环境中（地铁、农村、飞机上）。同时，幼儿有强烈的「重复听同一故事」的行为特征，同一页绘本可能被反复拍照朗读。

核心需求：
1. **离线可用**：已读过的页面在无网络时仍可朗读；
2. **避免重复请求**：相同页面不重复调用 OCR/TTS API，降低成本和延迟；
3. **存储受限**：微信小程序本地存储有限制（最大 200 MB），需要合理控制缓存量；
4. **简单可靠**：缓存策略不能过于复杂，要保证在各类边界情况下的稳定性。

### 3.2 备选方案

#### 方案 A：无缓存，每次联网请求

**优点：**
- 实现最简单，无缓存管理逻辑；
- 永远获取最新数据；

**缺点：**
- 离线完全不可用；
- 重复拍同一页时每次都调用 API，成本高、延迟大；
- 幼儿使用习惯（反复听）导致体验极差。

**结论**：不满足基本产品需求，直接排除。

#### 方案 B：永久全量缓存

**优点：**
- 所有已读页面永久保存，任何时候可重听；

**缺点：**
- 缓存无上限，长期使用后占用存储空间过大；
- 微信小程序本地存储上限 200 MB，全量缓存可能触发系统清理；
- 用户手机存储不足时体验恶化；
- 管理复杂，需处理存储满时的各种异常情况。

#### 方案 C：优先联网 + 离线缓存 20 页 LRU（**最终选择**）

**优点：**
- 缓存上限固定（20 页），存储占用可控（约 10–20 MB）；
- LRU 策略自动保留最近使用的页面，符合幼儿重复听同一绘本的行为模式；
- 离线时缓存页面可用，满足核心离线需求；
- 联网时优先获取最新 OCR 结果，保证准确性；
- 实现简单，边界情况易处理。

**缺点：**
- 超过 20 页后，旧页面会被淘汰，需要重新联网识别；
- 对于书页很多的绘本（>20 页），无法做到全书离线。

#### 方案 D：优先联网 + 离线缓存 50 页 LRU

**优点：**
- 缓存更多页面，适合页数多的绘本；

**缺点：**
- 存储占用约 25–50 MB，对手机存储有一定压力；
- 绝大多数儿童绘本页数 ≤ 40 页，且幼儿常用绘本通常 10–20 本，常看的页数有限；
- 超出的收益边际递减，但存储成本线性增加。

### 3.3 决定

> ✅ **采用方案 C：优先联网 + 离线缓存 20 页 LRU**

**缓存策略细节：**

```
策略：优先联网，降级离线
缓存容量：20 页（约 10–20 MB 存储）
淘汰算法：LRU（最近最少使用）
缓存内容：图片哈希 + OCR 文字结果 + TTS 音频文件
缓存键：图片内容 MD5 哈希（与文件名无关）
持久化：微信小程序 FileSystem（跨会话持久）

访问逻辑：
  有网络 → 优先调用 API → 结果存入缓存
  无网络 → 查询缓存 → 命中则播放，未命中则提示用户
  弱网（>5s 超时） → 先查缓存 → 命中则播放缓存，否则继续等待/提示
```

### 3.4 决策理由

1. **20 页容量符合实际使用场景**：经过用户研究，家长和幼儿的「活跃绘本」通常为 3–5 本，每次阅读集中在 3–8 页内。LRU 策略会自动保留最近反复听的页面，20 页能覆盖 90% 的高频使用场景。

2. **存储可控是硬约束**：微信小程序本地存储有上限，且用户手机存储空间参差不齐。20 页缓存约占用 10–20 MB，即使存储有限的低端机型也能承受，不会触发系统警告。

3. **LRU 与幼儿行为高度契合**：幼儿喜欢反复听最近听过的故事。LRU 算法会保留最近访问的页面，恰好与「最近在看的绘本」对应，淘汰久不使用的页面，符合直觉。

4. **「优先联网」保证准确性**：OCR 结果与照片光线、角度有关，每次联网可获取最优结果。仅在无网络时降级到缓存，保证了在线体验的质量上限。

5. **与 TTS 成本控制协同**：缓存命中时直接播放本地音频，不重复调用讯飞 TTS API。按照每页 50 字符估算，20 页缓存覆盖的重复请求节省约 60–70% 的 API 调用量。

6. **50 页方案收益边际递减**：对比方案 D，将缓存从 20 页扩展到 50 页，存储成本增加 2.5 倍，但用户感知到的离线体验提升有限（绝大多数单次使用不超过 20 页）。20 页是成本与体验的最优平衡点。

### 3.5 影响

| 维度 | 影响说明 |
|---|---|
| **用户体验** | ✅ 离线场景下常用绘本可正常朗读，解决地铁/农村场景痛点 |
| **性能** | ✅ 缓存命中响应时间 ≤ 0.5 秒，远优于在线识别的 3 秒 |
| **API 成本** | ✅ 减少 60–70% 的 OCR/TTS 重复调用，显著降低运营成本 |
| **存储占用** | ✅ 上限约 20 MB，对用户手机存储无明显影响 |
| **研发工作量** | LRU 缓存管理模块约 3–4 人天，含边界情况处理 |
| **后续迭代** | 📋 Phase 2 可根据数据分析评估是否将缓存上限调整为 30 页；Phase 3 可探索「绘本级缓存」（用户可手动缓存整本绘本） |
| **边界处理** | ⚠️ 需处理存储写入失败（磁盘满）、缓存文件损坏等异常情况，降级为联网模式 |
